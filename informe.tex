\section{Introducción}

\subsection{Contexto del problema}
Un centro de salud regional necesita modernizar su infraestructura, pasando de un sistema completamente local a uno distribuido. La información clínica de los pacientes debe estar siempre disponible, ya que una falla en el acceso a las fichas puede tener consecuencias directas en la atención médica.

Este proyecto toma como base la propuesta arquitectónica desarrollada en el laboratorio anterior y la lleva a un siguiente nivel, simulando el funcionamiento de dos áreas del centro de salud: las Estaciones Médicas y las Terminales Administrativas. Para esto se desplegaron contenedores Docker con servidores de aplicación, bases de datos con replicación y un proxy inverso Nginx, utilizando WebSockets como mecanismo de comunicación en tiempo real entre los distintos componentes.

\subsection{Objetivo general}
Diseñar y desplegar una arquitectura distribuida que simule la infraestructura tecnológica de un centro de salud, garantizando la disponibilidad, sincronización y seguridad de la información clínica entre sus distintas áreas. La solución debe ser capaz de soportar la comunicación en tiempo real entre los componentes del sistema, mantener la persistencia de los datos ante posibles fallos y operar de forma segura controlando el acceso al tráfico entrante.

\subsection{Objetivo específico}
\begin{itemize}
    \item Implementar la división de la infraestructura en subredes locales y en la nube para aislar el tráfico clínico local de las admisiones administrativas generales.
    \item Habilitar un canal de replicación lógica bidireccional asíncrona entre las bases de datos para sincronizar fichas clínicas en segundo plano.
    \item Desarrollar backends en Node.js usando Socket.io v4.7.5 para gestionar la comunicación clínica en tiempo real mediante eventos sobre WebSockets.
    \item Configurar Nginx para actuar como proxy de borde, administrando los upgrades de protocolo HTTP a WebSocket y protegiendo el sistema con límites de peticiones.
\end{itemize}

\section{Diseño arquitectónico y topología de red}

\subsection{Descripción de la solución híbrida}
Para la simulación del hospital optamos por un esquema de red híbrido que separa los servicios según el entorno donde operan:
\begin{itemize}
    \item \textbf{Red local del hospital (\texttt{hospital\_net} en VM1):} Conecta directamente las Estaciones Médicas (\texttt{app-estaciones}) y su base de datos local (\texttt{db-local}). El objetivo es asegurar que si el hospital pierde la conexión WAN (Internet), los médicos puedan seguir operando consultando y modificando la base de datos local, la cual sirve como caché. Esto reduce la latencia interna a valores casi nulos.
    \item \textbf{Red de la nube (\texttt{cloud\_net} en VM2):} Conecta el Módulo de Terminales Administrativas (\texttt{app-terminales}) y la base de datos core central (\texttt{db-nube}). Las admisiones se procesan directamente en este entorno remoto para no saturar los recursos locales del hospital y consolidar las admisiones en un repositorio central.
\end{itemize}

\subsection{Zona Desmilitarizada (DMZ)}
Para evitar la exposición directa a Internet de los servidores de aplicación y de base de datos internos, creamos una red de Zona Desmilitarizada (\texttt{dmz\_net}).
\begin{itemize}
    \item \textbf{El proxy como intermediario:} El contenedor \texttt{nginx-proxy} (VM3) es el único con presencia en las tres redes (\texttt{hospital\_net}, \texttt{cloud\_net} y \texttt{dmz\_net}). Actúa como el único punto de entrada, ruteando el tráfico externo que entra por el puerto 80/443 hacia las aplicaciones en los puertos 8001 y 8002.
    \item \textbf{Túnel externo sin apertura de puertos:} En lugar de abrir puertos públicos en la máquina virtual de GCP, el contenedor \texttt{cloudflare-tunnel} opera dentro de \texttt{dmz\_net} y establece un canal de salida seguro que asocia el subdominio \texttt{osdsp3.epistia.cl} hacia Nginx.
    \item \textbf{Canal de replicación interno:} La comunicación PostgreSQL para la sincronización entre \texttt{db-local} y \texttt{db-nube} se realiza por el puerto 5432 utilizando el direccionamiento de la red interna de GCP (resolviendo por IPs privadas \texttt{10.128.0.10} y \texttt{10.128.0.20} mediante directivas \texttt{extra\_hosts} en Docker Compose), manteniendo este tráfico totalmente invisible para usuarios externos.
\end{itemize}

\section{Detalle de componentes y módulos de aplicación}

\subsection{Módulo de Estaciones médicas}
\begin{itemize}
    \item \textbf{Desarrollo Backend WebSocket:} El servidor del módulo de Estaciones Medicas fue desarrollado en Node.js utilizando la librería Socket.io v4.7.5, levantando el servicio en el puerto 8001 bajo el sub-path \texttt{/ws-medicas/}. Esta configuración permite al proxy inverso Nginx identificar y enrutar correctamente el tráfico hacia este módulo de forma independiente al módulo administrativo.
    \item \textbf{Eventos principales de comunicación bidireccional:}
    \begin{itemize}
        \item \texttt{consultar\_paciente}: recibe un RUT como parámetro, consulta la base de datos local y retorna la ficha clínica activa del paciente.
        \item \texttt{actualizar\_diagnostico}: recibe el identificador UUID del registro junto al nuevo diagnóstico, ejecutando una operación UPDATE directa sobre la tabla \texttt{fichas\_pacientes}.
    \end{itemize}
    \item \textbf{Manejo de caché de alta demanda:} La base de datos local (\texttt{db-local}) corre en un contenedor PostgreSQL 16 dentro de la red \texttt{hospital\_net}, aislada de la base de datos de la nube. Actúa como caché de alta demanda, almacenando las fichas clínicas activas de los pacientes en atención, permitiendo que las estaciones médicas operen con baja latencia sin depender de la disponibilidad de la conexión externa.
    \item \textbf{Simulación del flujo clínico:} El flujo comienza cuando el módulo administrativo admite un paciente, generando un UUID y almacenando su ficha en la base de datos de la nube. Una vez replicado el registro hacia \texttt{db-local}, el médico puede consultar la ficha ingresando el RUT del paciente desde su estación. Con los datos en pantalla, el médico actualiza el diagnóstico usando el UUID del registro, quedando el cambio persistido localmente. Para simular este flujo, se precargaron tres fichas de prueba en \texttt{db-local} al inicializar el contenedor, representando casos clínicos reales como controles crónicos, urgencias y altas médicas.
\end{itemize}

\subsection{Módulo de Terminales Administrativas}
\begin{itemize}
    \item \textbf{Desarrollo de Backend de comunicación síncrona:} Se programa el servidor Node.js usando Socket.io sobre el protocolo WebSocket, configurando el puerto 8002 y el sub-path \texttt{/ws-administrativas} para el enrutamiento del proxy inverso.
    \item \textbf{Lógica de negocio e inyección de metadatos:} Se implementa el manejador de eventos \texttt{admitir\_paciente}, el que gestiona la concurrencia, mitiga colisiones relacionales mediante identificadores universales y añade el metadato crítico \texttt{origen\_registro: 'nube'} para activar las reglas de replicación selectiva condicional de PostgreSQL 16.
\end{itemize}

\subsection{Módulo Transversal de Infraestructura y Orquestación}
Para desplegar los servicios de forma controlada usamos archivos \texttt{docker-compose.yml} separados para cada nodo.
\begin{itemize}
    \item \textbf{Habilitación de logs de replicación:} Las bases de datos se levantan con parámetros específicos para habilitar la extracción de logs de transacciones en modo lógico:
    \begin{verbatim}
    postgres -c wal_level=logical -c max_connections=100
             -c max_replication_slots=10 -c max_wal_senders=10
    \end{verbatim}
    Esto le permite a PostgreSQL estructurar el stream de replicación como cambios atómicos sobre filas en lugar de duplicar los bloques físicos del disco.
    \item \textbf{Persistencia y recuperación:} Los datos se guardan en volúmenes nombrados de Docker (\texttt{db\_local\_data} y \texttt{db\_cloud\_data}). De esta manera, si detenemos o recreamos los contenedores, la información de los pacientes y los metadatos de replicación (suscripciones y slots) no se pierden.
    \item \textbf{Control de ejecución:} Usamos \texttt{restart: always} para que Docker intente reanudar automáticamente los contenedores si el nodo de GCP se cae o se reinicia.
\end{itemize}

\subsection{Módulo de Seguridad Perimetral y Capa Cliente}
\begin{itemize}
    \item \textbf{Nginx como Proxy Inverso:} Nginx actúa como puerta de enlace de red. Aplicamos \texttt{server\_tokens off} para ocultar su versión. Configuramos rate limiting usando \texttt{limit\_req\_zone} con una zona de almacenamiento en memoria de 10 MB para soportar una tasa base de 10 peticiones por segundo por IP, permitiendo ráfagas transitorias de hasta 15 peticiones sin retardo inicial (\texttt{burst=15 nodelay}) en las rutas de WebSocket.
    \item \textbf{Negociación y timeouts de WebSockets:} Para que las conexiones de Socket.io se establezcan correctamente, configuramos Nginx para interceptar la negociación y actualizar el protocolo HTTP/1.1 a WebSocket:
    \begin{verbatim}
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection $connection_upgrade;
    \end{verbatim}
    Ajustamos \texttt{proxy\_read\_timeout} y \texttt{proxy\_send\_timeout} a 24 horas (\texttt{86400s}) para evitar que Nginx corte la conexión de las estaciones médicas por falta de actividad durante el día.
    \item \textbf{Aplicación de Simulación Frontend:} Diseñamos una interfaz web de una sola página (SPA) responsiva. La aplicación lee el rol seleccionado (Médico, Enfermero o Administrativo) y muestra únicamente el panel clínico o el panel administrativo que corresponde. La comunicación utiliza Socket.io con transporte forzado a WebSockets, gestiona reconexiones automáticas si el servidor se cae, e indica el estado del canal mediante un indicador circular que cambia de color dinámicamente.
\end{itemize}

\section{Modelo de datos y estrategia de sincronización distribuida}

\subsection{Esquema unificado de persistencia}
Tanto en la base local como en la nube compartimos la misma estructura para la tabla \texttt{fichas\_pacientes}:

\begin{lstlisting}[language=SQL]
CREATE TABLE IF NOT EXISTS fichas_pacientes (
    id UUID PRIMARY KEY,
    rut VARCHAR(20) NOT NULL,
    nombre VARCHAR(100) NOT NULL,
    diagnostico TEXT,
    origen_registro VARCHAR(50) NOT NULL,
    fecha_actualizacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
ALTER TABLE fichas_pacientes REPLICA IDENTITY FULL;
\end{lstlisting}

Usamos llaves primarias de tipo UUID para que al ingresar pacientes de forma paralela en la nube y en el hospital local, no ocurran colisiones de identificadores numéricos tradicionales al replicar. Además, activamos la directiva \texttt{REPLICA IDENTITY FULL} para obligar a PostgreSQL a incluir el valor anterior de todas las columnas de la fila en su registro WAL. Esto es necesario para que el sistema de replicación identifique la fila exacta que debe modificar cuando ocurre un UPDATE.

\subsection{Mecanismo de replicación lógica}
Decidimos usar replicación lógica en lugar de física por una razón clave: la replicación física obligaría a que la base de datos local del hospital (\texttt{db-local}) estuviese en modo de solo lectura. Con la replicación lógica podemos tener un esquema activo-activo, donde ambos nodos leen y escriben datos de manera simultánea. 

Además, este método asíncrono transmite únicamente los cambios a nivel de fila sobre la tabla \texttt{fichas\_pacientes}, optimizando el ancho de banda y garantizando la consistencia eventual sin importar si hay caídas temporales de la conexión WAN.

\subsection{Mitigación de bucles infinitos de datos}
El problema de replicar de forma bidireccional activa-activa es el bucle circular: los cambios recibidos por replicación se vuelven a enviar al nodo de origen. Para evitar este problema implementamos dos capas de seguridad:

\begin{enumerate}
    \item \textbf{Filtros en Publicaciones:} Restringimos el flujo de datos usando la cláusula \texttt{WHERE} de las publicaciones sobre la columna \texttt{origen\_registro}:
    \begin{lstlisting}[language=SQL]
    -- En db-local (Hospital)
    CREATE PUBLICATION pub_local_a_nube FOR TABLE fichas_pacientes WHERE (origen_registro = 'local');

    -- En db-nube (Nube)
    CREATE PUBLICATION pub_nube_a_local FOR TABLE fichas_pacientes WHERE (origen_registro = 'nube');
    \end{lstlisting}
    Así, cada nodo solo exporta datos generados en su propia red local.
    \item \textbf{Suscripciones con exclusión de origen (\texttt{origin = none}):}
    Al definir las suscripciones a nivel SQL, agregamos el parámetro \texttt{origin = none}:
    \begin{lstlisting}[language=SQL]
    CREATE SUBSCRIPTION sub_desde_nube CONNECTION '...' 
    PUBLICATION pub_nube_a_local WITH (copy_data = false, origin = none);
    \end{lstlisting}
    Esto le indica a la base de datos que ignore cualquier transacción que no se haya originado localmente en el publicador. Esto rompe de manera definitiva el ciclo circular de actualizaciones de registros.
\end{enumerate}

\section{Escenarios de prueba y validación del sistema}

\subsection{Escenario 1: Operación normal en tiempo real (WebSockets)}
\textbf{Verificación:} Con Nginx operando, iniciamos sesión como Administrativo e ingresamos una ficha (RUT: \texttt{77777777-7}, Nombre: \texttt{Clara Campo}). Al enviar el formulario, el cliente Socket.io despachó la petición hacia el endpoint \texttt{/ws-administrativas}. El servidor de admisión procesó el evento \texttt{admitir\_paciente}, insertó la ficha en \texttt{db-nube} y retornó el evento \texttt{paciente\_admitido\_confirmado} actualizando el feed del dashboard en menos de 50 milisegundos sin requerir la recarga del navegador.

\subsection{Escenario 2: Sincronización y replicación exitosa}
\textbf{Verificación:} Comprobamos que tras ingresar a la paciente Clara Campo en la nube (con origen marcado como \texttt{'nube'}), el registro se replicó de forma asíncrona hacia la base del hospital. Al ingresar al sistema con el rol de Médico en una Estación y buscar el RUT \texttt{77777777-7}, la aplicación recuperó el registro cargándolo directamente desde \texttt{db-local}. Posteriormente, ingresamos un diagnóstico y presionamos "Guardar". La estación ejecutó la actualización en la base local, la cual se transmitió con éxito de vuelta hacia la base de datos de la nube central.

\subsection{Escenario 3: Resiliencia ante caídas de conectividad}
\textbf{Verificación:} Detuvimos el contenedor de base de datos de la nube (\texttt{db-nube}) para simular un corte en el enlace de Internet. En la Estación Médica del hospital, realizamos modificaciones diagnósticas de forma local. Comprobamos que el sistema local continuó respondiendo y registrando las consultas sobre \texttt{db-local}. Tras restablecer la red e iniciar nuevamente el contenedor de la nube, la suscripción en PostgreSQL reanudó de manera automática la conexión de réplica, consumiendo el backlog almacenado en los logs de transacciones del WAL sin pérdida de registros clínicos.

\section{Conclusiones}
\begin{itemize}
    \item La utilización de WebSockets proxyados por Nginx permite reducir al mínimo la latencia del sistema clínico de simulación en comparación con esquemas REST tradicionales. Al eliminar el sobrecosto de cabeceras en peticiones continuas, optimizamos el rendimiento general del tráfico de red en la infraestructura.
    \item La replicación lógica activa-activa condicional junto al parámetro \texttt{origin=none} provee una solución robusta para mantener la consistencia eventual. Esto asegura la continuidad operacional del hospital regional, permitiéndole trabajar localmente con baja latencia y de manera autónoma ante fallos de conexión WAN.
\end{itemize}
